5 MCP Security Mistakes Every AI Agent Builder Makes

February 25, 2026 · 8 min read

The Model Context Protocol is transforming how AI agents interact with external services. But with great power comes great attack surface. Most MCP deployments we've seen have at least three of these five security gaps.

Here's what's going wrong — and how to fix each one.

1 Hardcoding API Keys in MCP Server Config

This is the most common mistake by far. The typical MCP setup guide tells you to put your API key directly in the server configuration:

{
  "mcpServers": {
    "my-api": {
      "command": "node",
      "args": ["server.js"],
      "env": {
        "STRIPE_KEY": "sk_live_abc123...",
        "OPENAI_KEY": "sk-def456..."
      }
    }
  }
}

The problem? That config file is readable by every process on your machine. It's probably in a git repo somewhere. And worst of all — the MCP server (and the AI agent using it) has your raw API key in its process environment.

One prompt injection and that key is exfiltrated in a tool call to an attacker-controlled URL.

✅ Fix: Use a secrets manager that injects credentials at request time, not at startup. The agent should never have the raw key in memory. Janee's proxy approach passes requests through a credential-injecting proxy — the agent sees a baseUrl but never the key itself.

2 No Audit Trail for Agent API Calls

When your AI agent makes 47 API calls in a session, can you tell me:

  • Which endpoints were hit?
  • What HTTP methods were used?
  • What was the response status?
  • Was any data modified or deleted?

Most MCP setups have zero visibility here. The agent calls a tool, the tool hits an API, and nobody knows what happened unless you go digging through provider dashboards after the fact.

This matters most when things go wrong. A billing spike, unexpected data changes, or a rate limit hit — without audit logs, you're flying blind.

✅ Fix: Route agent API traffic through a logging proxy. Every request should be captured with timestamp, method, path, and response status. Janee's audit log records all of this automatically — janee sessions shows exactly what each agent session did.

3 Giving Agents Full API Access (No Policies)

Your coding agent needs to read GitHub repos. So you give it a GitHub token with repo scope. Now it can also:

  • Delete repositories
  • Push to main
  • Create webhooks
  • Modify branch protection rules
  • Access private repos it doesn't need

This is the principle of least privilege, and almost nobody applies it to AI agents. The common excuse: "The token scope is as narrow as I can make it." But API providers give coarse-grained scopes. repo means everything.

✅ Fix: Layer request-level policies on top of API tokens. Even if the underlying token has broad permissions, your proxy should enforce rules like "only GET requests to /repos/myorg/*" or "no DELETE methods." Janee's policy engine lets you define allow/deny rules per service at the HTTP method and path level.

4 No Session Timeouts or Kill Switch

Once an agent has API access, how long does it keep it? In most MCP setups: forever. The API key is in the config, so the agent has permanent access until you manually rotate the key.

This is especially dangerous for autonomous agents that run unattended. If an agent enters a bad loop and starts making hundreds of API calls per minute, you need a way to cut it off immediately — not in 30 minutes when you notice your bill.

✅ Fix: Use session-based access with TTLs. Agent gets a time-limited session (e.g., 1 hour). When the session expires, access is gone — no key rotation needed. Janee supports session TTLs with instant revocation: janee sessions revoke <id> kills access in under a second.

5 Running MCP Servers Over HTTP Without Authentication

The MCP spec supports both stdio (local) and SSE/Streamable HTTP (remote) transports. When you move to HTTP-based transport for remote agents, you need authentication on the MCP server itself.

We've seen MCP servers deployed on internal networks with no auth — just an open HTTP endpoint that any process can connect to. This means any compromised service on that network can control your AI agent's tools.

The MCP spec now includes OAuth 2.1 authorization, but many server implementations haven't caught up.

✅ Fix: For local agents, prefer stdio transport (no network exposure). For remote agents, always require authentication on your MCP server. If your server doesn't support auth natively, put it behind an authenticating reverse proxy.

The Pattern

Notice the common thread? All five mistakes come from treating AI agents like trusted internal services. They're not. They're executing untrusted instructions (user prompts) with access to your real infrastructure.

The security model for AI agents should look more like a zero-trust network:

  • Never trust the agent with raw credentials
  • Log everything
  • Enforce least-privilege at the request level
  • Time-limit all access
  • Authenticate all connections
